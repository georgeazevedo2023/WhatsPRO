## 1. Agente IA — Regras de Processamento

### 1.1 Debounce e Agrupamento de Mensagens
| Regra | Valor | Configuravel |
|-------|-------|-------------|
| Janela de debounce padrao | 10 segundos | Sim (agent.debounce_seconds) |
| Mecanismo de fila | RPC `append_ai_debounce_message` (atomico) | Nao |
| Fallback se RPC falhar | Merge in-memory (buildLegacyQueueUpdate) | Nao |
| Retry em erro 5xx | 1 retry apos 2s | Nao |
| Typing indicator | Enviado imediatamente ao receber primeira msg | Nao |

### 1.2 Greeting (Saudacao)
| Regra | Detalhe |
|-------|---------|
| Deteccao de saudacao | 28+ variantes PT-BR/EN (oi, ola, bom dia, etc.) |
| Lead novo (nunca interagiu) | Envia `greeting_message` estatica |
| Lead retornando (interagiu >24h atras) | Envia `returning_greeting_message` com `{nome}` substituido pelo `lead_profile.full_name` |
| Returning greeting default | "Ola {nome}! Que bom te ver aqui de novo. Em que posso te ajudar hoje?" |
| Msg e APENAS saudacao | Envia greeting, pula chamada LLM |
| Dedup de greeting | RPC `try_insert_greeting` (advisory lock atomico) |
| Cutoff de interacao recente | 24 horas (verifica ai_agent_logs.created_at) |
| Greeting no Playground | Injetado como model message em geminiContents (nao no system prompt) |
| Personalizacao | Greeting DEVE usar nome do lead quando conhecido |

### 1.3 Handoff (Transbordo para Humano)
| Regra | Detalhe |
|-------|---------|
| Triggers por regex | 5 padroes: "encaminhar", "transferir", "atendente humano", "falar com vendedor" (com negative lookahead) |
| Gate de interacao | Handoff trigger so dispara se agente interagiu nas ultimas 24h |
| Mensagem de transicao | Customizavel (default: "So um instante que vou te encaminhar...") |
| Pos-handoff | status_ia muda para 'shadow' |
| Limite de mensagens | Auto-handoff apos 8 msgs incoming (configuravel: agent.max_lead_messages) |
| Reset do contador | Quando tag "ia_cleared:TIMESTAMP" adicionada |
| Tool handoff_to_human | Envia 1 msg + quebra o loop LLM (sem texto duplicado) |
| Deteccao implicita | Verificada ANTES de enviar resposta do agente |

### 1.4 Shadow Mode
| Regra | Detalhe |
|-------|---------|
| Constante | STATUS_IA.SHADOW = 'shadow' |
| Comportamento | IA extrai dados (tags, labels, lead profile) SEM responder ao contato |
| Tools permitidas | set_tags, update_lead_profile (nao envia msgs) |
| Temperature | 0.2 (deterministico) |
| Ativacao | 1) Handoff trigger, 2) Switch manual pelo usuario, 3) Tool call explicito |

### 1.5 Horario Comercial
| Regra | Detalhe |
|-------|---------|
| Formato | start/end como "HH:MM" (24h) |
| Timezone | Sempre "America/Sao_Paulo" |
| Fora do horario | Envia `out_of_hours_message`, pula processamento LLM |
| Wraparound | Suporta horarios noturnos (ex: 22:00-06:00) |

### 1.6 Fluxo SDR (Qualificacao)
| Regra | Detalhe |
|-------|---------|
| Estrategia Zero-Call | Max 3 perguntas antes de busca/handoff |
| Termo generico (ex: "tinta") | Fazer perguntas para refinar |
| Termo especifico (ex: "Tinta Coral 18L") | Busca imediata no catalogo |
| 0 resultados | Valorizacao + handoff (NUNCA dizer "nao encontrado") |
| 1 resultado | Enviar midia + copy |
| 2-5 resultados | Enviar carrossel |
| 6-10 resultados | 2 carrosseis em batches |
| 10+ resultados | Pergunta final ou handoff |
| Sub-agentes | 5 modos: SDR, Sales, Support, Scheduling, Handoff |

### 1.7 Bloqueio de IA
| Regra | Detalhe |
|-------|---------|
| Bloqueio por contato | `contact.ia_blocked_instances` inclui instance_id atual → skip |
| Bloqueio por numero | `contact.phone` match em `agent.blocked_numbers` → skip |
| IA desligada | status_ia='desligada' → skip completo |
| Clear context | Reseta status_ia='ligada' + limpa ia_blocked_instances + tag ia_cleared |

### 1.8 LLM Providers e Fallback Chain
| Prioridade | Provider | Modelo | Uso |
|------------|----------|--------|-----|
| 1 (primario) | OpenAI | gpt-4.1-mini (configuravel) | Agente principal |
| 2 (fallback) | Gemini | gemini-2.5-flash | Fallback do agente |
| 1 (carousel) | Groq | Llama 3.3 | Copy de carrossel |
| 2 (carousel) | Gemini | 2.5 Flash | Fallback carousel |
| 3 (carousel) | Mistral | Small | Segundo fallback |
| 4 (carousel) | Estatico | Templates hardcoded | Fallback final |

**Timeouts por provider carousel**: 2s cada
**Carousel card 1**: code-generated (titulo + preco, sem LLM)
**Carousel cards 2-N**: AI-generated com chain acima
**Cache**: 24h TTL, 200 items max, LRU eviction

### 1.9 Circuit Breaker
| Provider | Threshold | Reset |
|----------|-----------|-------|
| OpenAI | 3 falhas | 30s |
| Gemini | 3 falhas | 30s |
| Groq | 3 falhas | 20s |
| Mistral | 3 falhas | 20s |
| UAZAPI | 5 falhas | 60s |

**Estados**: CLOSED → OPEN (apos threshold) → HALF_OPEN (apos reset) → CLOSED (apos sucesso)
**Half-Open**: 1 request de prova permitido

### 1.10 TTS (Text-to-Speech)
| Regra | Detalhe |
|-------|---------|
| Provider | Gemini 2.5 Flash Preview TTS |
| Vozes | 6 opcoes Kore (Kore = default) |
| Formato | WAV (24kHz, 16-bit PCM) |
| Envio UAZAPI | Tipo "ptt" (Push-to-Talk) |
| Fallback | Se TTS falhar, cai para texto |
| Delay de digitacao | ~40ms/char, min 1s, max 5s |

### 1.11 Tools do Agente (8 total)
| Tool | Funcao | Side Effects |
|------|--------|-------------|
| search_products | Busca catalogo (ILIKE + word fallback) | Nenhum |
| send_carousel | Carrossel multi-card WhatsApp | Envia msg via UAZAPI |

## 2. Webhook WhatsApp (UAZAPI)

### 2.1 Fluxo de Mensagem
1. UAZAPI envia POST para `/whatsapp-webhook`
2. Valida WEBHOOK_SECRET (opcional mas recomendado)
3. Detecta formato: UAZAPI nativo vs raw agent output vs status_ia-only
4. Grupos: skip total (return 200)
5. Resolve instancia (inbox_id → instanceName → owner_jid → owner phone)
6. Parallel: media download + dedup check + contact upsert
7. Cria/atualiza conversa (status open/pending)
8. Insere mensagem com dedup (external_id UNIQUE)
9. Fire-and-forget: lead_database auto-add, UTM attribution, follow-up status
10. Se incoming + nao fromMe + nao audio + status_ia != DESLIGADA → trigger debounce

### 2.2 Tipos de Mensagem Suportados
| Tipo | Normalizacao | Tratamento Especial |
|------|-------------|-------------------|
| Texto | "text" | Direto |
| Imagem | "image" | Media URL extraction |
| Video | "video" | Media URL extraction |
| Audio | "audio" | Transcricao async (Gemini ASR, 90s timeout) |
| Documento | "document" | PDF extraction |
| Sticker | "sticker" | Armazena tipo |
| Contato (vCard) | "contact" | JSON serializado em media_url |
| Resposta botao | "text" | selectedButtonText extraido |
| Resposta lista | "text" | listResponse.title/id extraido |

### 2.3 Normalizacao de Campos UAZAPI
- Campos podem vir PascalCase ou camelCase (inconsistente)
- Sempre tratar ambas variantes: EventType/eventType, message/Message
- Content pode ser string OU objeto (message.content.text)
- Media URL: fileURL > mediaUrl > content.URL > content.url
- External ID: se contem ":", usa ultimo segmento

### 2.4 Download de Midia
| Regra | Detalhe |
|-------|---------|
| Endpoint | POST /message/download (via instance token) |
| Audio | generate_mp3=true, prefere raw link (ogg/opus) |
| URL fallback | data.link → data.url → data.fileUrl → data.fileURL → data.mp3Link |
| Persistencia | URLs armazenadas diretamente no DB (SEM re-upload para Storage) |

## 3. Seguranca e Acesso

### 3.1 Papeis (RBAC)
| Role | Permissoes | Escopo |
|------|-----------|--------|
| super_admin | Acesso total, gerencia instancias/inboxes/usuarios | Global |
| gerente | Gerencia equipe em inboxes atribuidas, CRM, leads | Por inbox |
| user (agente) | Atende conversas em inboxes atribuidas | Por inbox |

### 3.2 Padroes RLS (Row Level Security)
| Padrao | Implementacao | Tabelas |
|--------|--------------|---------|
| Owner-based | `auth.uid() = user_id` | broadcast_logs, templates, lead_databases |
| Inbox access | `has_inbox_access(auth.uid(), inbox_id)` | conversations, contacts, messages |
| Super admin | `is_super_admin(auth.uid())` | Todas (bypass) |
| Department | `EXISTS (department_members WHERE user_id = auth.uid())` | conversations |
| Kanban board | `can_access_kanban_board(auth.uid(), board_id)` | kanban_cards, columns, fields |
| Kanban card | `can_access_kanban_card(auth.uid(), card_id)` | kanban_card_data |

### 3.3 JWT e Autenticacao
| Regra | Detalhe |
|-------|---------|
| Edge functions com JWT | 20 de 27 (verify_jwt=true) |
| Sem JWT (webhooks) | whatsapp-webhook, fire-outgoing-webhook, go, health-check |
| Funcoes internas | ai-agent, ai-agent-debounce: verify_jwt=false (chamadas inter-funcao) |
| Token UAZAPI | NUNCA exposto ao frontend; resolvido server-side no uazapi-proxy |
| API keys | Armazenadas no Supabase Vault |
| Audit log | admin_audit_log (imutavel) + log_admin_action() RPC |

### 3.4 Webhook Security
| Regra | Detalhe |
|-------|---------|
| WEBHOOK_SECRET | Opcional (log WARNING se ausente) — UAZAPI nao suporta auth headers |
| Validacao | x-webhook-secret header OU Authorization Bearer |
| Producao | Recomendado mas NAO obrigatorio |

## 4. Broadcast

### 4.1 Limites
| Limite | Valor |
|--------|-------|
| Max destinatarios contatos | 500 por broadcast |
| Max destinatarios grupos | 50 por broadcast |
| Max cards carrossel | 10 por mensagem |
| Max tamanho audio | 12 MB |
| Max tamanho CSV | 10 MB |
| Max linhas CSV | 50.000 |

### 4.2 Delays Anti-Spam
| Opcao | Range |
|-------|-------|
| Sem delay | 0s |
| Delay curto | 5-10s |
| Delay longo | 10-20s |
| Base delay (scheduled) | 350ms entre msgs do batch |

### 4.3 CSV Sanitizacao
- Strips: `=`, `+`, `@`, `-` no inicio de celulas (prevencao injection)
- Validacao de telefone: >= 10 caracteres

# DB Reset 2026-05-19

> Limpeza completa de dados operacionais (conversas, mensagens, leads, contatos, bases) **antes de cadastrar uma nova instância** — para garantir zero cruzamento de dados entre a instância antiga (Eletropiso) e a nova.

## Contexto

Usuário decidiu cadastrar uma nova instância produtiva e pediu reset total dos dados operacionais para evitar cruzamento (ex: contatos antigos da Eletropiso interagindo com a nova instância via `contacts.phone` reusado, conversas órfãs, logs de IA cruzados). Decisão tomada com escopo **total** (todas as instâncias) e **sem backup**.

## Operação executada

**Comando único** — `TRUNCATE ... RESTART IDENTITY CASCADE` em transação:

```sql
BEGIN;
TRUNCATE TABLE
  public.contacts, public.conversations, public.conversation_messages,
  public.conversation_labels, public.lead_profiles, public.lead_memory,
  public.lead_score_history, public.lead_databases, public.lead_database_entries,
  public.handoff_queue_events, public.poll_messages, public.poll_responses,
  public.ai_agent_logs, public.ai_agent_validations, public.ai_debounce_queue,
  public.conversion_funnel_events, public.flow_events, public.flow_followups,
  public.flow_security_events, public.flow_states, public.follow_up_executions,
  public.form_sessions, public.form_submissions, public.intent_detections,
  public.notification_log, public.pending_responses, public.shadow_extractions,
  public.validator_logs, public.utm_visits, public.bio_lead_captures,
  public.kanban_cards, public.assistant_conversations
RESTART IDENTITY CASCADE;
COMMIT;
```

Sem erros. Validado com `SELECT COUNT(*)` em todas as 19 tabelas auditadas — todas em 0.

## Riscos mitigados

- **FK CASCADE:** ~20 tabelas com FK para `contacts`/`conversations`/`lead_profiles` foram limpas no mesmo TRUNCATE (ordem explícita evita NO ACTION). Validado: nenhum erro.
- **Sequences:** `RESTART IDENTITY` reseta sequences — UUIDs gerados serão novos, sem colisão.
- **RLS:** operação rodou via MCP (postgres role) — não afetada por RLS.
- **Realtime:** publication `supabase_realtime` continua válida (TRUNCATE não derruba publication; só limpa linhas).

## Próximos passos pós-reset

1. Cadastrar nova instância (UI → Admin → Instâncias)
2. Conectar via UAZAPI (QR Code)
3. Criar inbox + department default
4. Atribuir usuários via `inbox_users`
5. Configurar `ai_agent_configs` para a nova instância
6. Validar com 1 mensagem teste

## Lição

Reset total é seguro **se feito em transação única com CASCADE explícito** — listar todas as tabelas filhas em vez de confiar só no CASCADE da FK evita deixar tabelas órfãs em estado inconsistente. Sempre auditar `information_schema.table_constraints` antes pra mapear FKs.
